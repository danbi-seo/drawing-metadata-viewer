## 2. 상태 관리 설계

### 2-1. Store 분리 원칙

Zustand 스토어를 **두 개**로 분리했습니다.

| Store | 역할 | 포함 상태 |
|-------|------|-----------|
| `drawingStore` | 도메인 : 상태 무엇을 보고 있는지 | 선택된 도면, 공종, 리비전, 오버레이 목록, 비교 모드 |
| `viewStore` | UI 상태 : 어떻게 보고 있는지 | 줌 배율, 패닝 오프셋, 사이드바 열림 여부, 폴리곤 표시 여부, 오버레이 투명도 |

**이유:**
- 도메인 상태와 UI 상태의 변경 주기가 다릅니다. 도면을 전환(`selectDrawing`)할 때 줌/패닝은 별도로 리셋되어야 하므로, 두 Store가 독립적으로 동작해야 합니다.
- 향후 URL 동기화나 서버 상태 도입 시, 도메인 상태만 별도로 정렬 가능합니다.

**고민했던 대안:**
- 단일 스토어: 처음에는 하나의 스토어에 모든 상태를 넣는 방식을 고려했습니다. 구현이 단순해지고 스토어 간 의존 관계가 없어진다는 장점이 있었습니다. 그러나 navigateToDrawing()처럼 도메인 상태 변경과 뷰 상태 리셋을 동시에 처리하는 작업이 많아질수록, 단일 스토어에서는 set() 한 번에 너무 많은 필드를 건드리는 코드가 반복됐습니다. 스토어 분리 후 useDrawingNavigation 훅에서 두 스토어를 조합하는 방식으로 정했습니다.

### 2-2. Cascade 초기화 전략

계층 구조를 내려갈 때 상위 선택이 변경되면 하위 상태를 전부 초기화합니다.

```
selectDrawing() 호출
  → selectedDisciplineKey = null
  → selectedRegionKey = null
  → selectedRevision = null
  → overlayDisciplineKeys = []
  → compareMode = false

selectDiscipline() 호출
  → selectedRegionKey = null
  → selectedRevision = null
  → overlayDisciplineKeys = []
  → compareMode = false
```

**이유:** 상위 컨텍스트가 바뀌었는데 하위 선택이 남아있으면 잘못된 데이터 조합이 렌더링될 수 있습니다.

**고민했던 대안:**
- 선택적 초기화: 같은 공종 이름이 다른 건물에도 존재할 경우 공종 선택을 유지하면 편할 수 있다는 생각으로 건물 전환 시 공종명이 동일하면 유지하는 방식을 시도해봤습니다. 그런데 공종이 같더라도 리비전 목록·이미지·폴리곤 좌표는 건물마다 완전히 다르기 때문에, 선택적 유지는 오히려 "공종은 선택되어 있는데 도면 이미지가 없는" 빈 화면 케이스를 만들었습니다. 결국 전체 초기화가 더 안전하다고 판단했습니다.

## 3. 데이터 레이어 설계

### 3-1. 파싱 vs 원본 사용

`metadata.json`의 원본 데이터를 컴포넌트에서 직접 사용하지 않고, `parser.ts`를 통해 **정규화(Normalization)** 후 사용합니다.

```
metadata.json (원본)
    ↓ parser.ts
NormalizedDiscipline / NormalizedRevision (정규화 타입)
    ↓ selectors.ts
컴포넌트
```

**정규화 시 처리하는 내용:**
- 리비전 배열을 날짜 기준 오름차순 정렬
- 가장 최신 리비전에 `isLatest: true` 마킹
- Region이 있는 공종과 없는 공종을 동일한 인터페이스로 통일

**이유:** 원본 JSON의 스펙이 바뀌거나 리비전 순서가 달라져도 parser.ts만 수정하면 됩니다. 초기에는 컴포넌트에서 revisions.sort(...)를 직접 호출하는 방식으로 작성했는데, 동일한 정렬 로직이 RevisionList와 useDrawingNavigation에 중복되면서 한 쪽을 수정할 때 다른 쪽을 놓치는 실수가 생겼습니다. 파싱 단계로 올린 이후 이 문제가 해결됐습니다.

### 3-2. 셀렉터 캐싱

`selectors.ts`의 `getDiscipline()` 함수는 내부적으로 `Map` 기반 메모이제이션을 적용했습니다.

```typescript
const disciplineCache = new Map<string, NormalizedDiscipline>()

// 키: "drawingId:disciplineKey" (예: "01:건축")
```

**이유:** 
- 폴리곤 레이어 계산(usePolygonLayers), 겹쳐보기 이미지 추출(DrawingCanvas), 사이드바 공종 목록(DisciplineTab) 등 여러 컴포넌트가 동일한 공종 데이터를 독립적으로 조회합니다. 메타데이터가 정적이기 때문에 캐시 무효화 전략 없이 단순 Map으로 충분하다고 판단했습니다.

## 4. 도면 렌더링 설계

### 4-1. Canvas vs HTML/SVG 선택

도면 이미지 렌더링에 `<canvas>` 대신 **HTML `<img>` + CSS transform** 방식을 선택했습니다.

| 항목 | `<canvas>` | `<img>` + CSS transform |
|------|-----------|------------------------|
| 줌/패닝 구현 | 매 프레임 재드로우 필요 | CSS transform으로 GPU 가속 |
| 다중 이미지 합성 | 직접 구현 필요 | `position: absolute` + `opacity` |
| 폴리곤 오버레이 | 캔버스에 직접 드로우 | SVG를 이미지 위에 절대 위치로 겹침 |
| 접근성 | 낮음 | `alt` 속성 지원 |


**이유:** 
- Canvas API를 먼저 시도했는데, 줌/패닝마다 clearRect → drawImage 사이클이 필요하고 고해상도 도면에서 프레임 드랍이 발생했습니다. <img> + CSS transform은 브라우저 합성 레이어를 활용하므로 CPU 부하 없이 부드러운 줌/패닝이 가능했습니다. 폴리곤을 SVG로 분리해서 이미지와 겹치는 방식도 DOM 구조가 명확해서 유지보수가 쉬웠습니다.

### 4-2. 폴리곤 좌표계

폴리곤 SVG는 이미지와 **동일한 좌표계**를 공유합니다.

```tsx
// SVG viewBox를 이미지 원본 크기와 동기화
<svg viewBox={`0 0 ${imgSize.w} ${imgSize.h}`}>
```

이미지가 CSS로 축소/확대되어도 SVG가 함께 스케일되어 좌표 보정 없이 폴리곤이 정렬됩니다.

**`polygonTransform` 적용 이유:**
도면 이미지 내에서 공종별 도면의 위치·스케일·회전이 다를 수 있기 때문에, 각 폴리곤 정점에 2D 변환 행렬(Rotation Matrix)을 적용하여 올바른 위치에 렌더링합니다.

```
rotated_x = x * scale * cos(θ) - y * scale * sin(θ)
rotated_y = x * scale * sin(θ) + y * scale * cos(θ)
```
**이유:** 
- CSS로 이미지가 축소·확대되어도 SVG가 함께 스케일되어, 폴리곤 좌표를 픽셀 단위로 별도 보정할 필요가 없습니다. 초기에는 이미지 표시 크기를 직접 계산해서 폴리곤 좌표에 배율을 곱하는 방식을 썼는데, 줌 배율이 바뀔 때마다 좌표를 재계산해야 해서 usePolygonLayers가 뷰 상태에 의존하게 되는 문제가 생겼습니다. viewBox 동기화 방식으로 변경 후 훅이 순수하게 도메인 상태만 바라보게 됐습니다.

### 4-3. 오버레이 합성 방식

다중 공종 겹쳐보기는 `mix-blend-mode: multiply`를 활용합니다.

```tsx
<img style={{ mixBlendMode: 'multiply', opacity: overlayOpacity }} />
```

- `multiply` 블렌드 모드: 두 이미지의 픽셀 값을 곱하여, 흰 배경 도면에서 선만 겹쳐 보이는 효과를 냅니다.
- 투명도 슬라이더(0~100%)로 오버레이 강도를 실시간 조절할 수 있습니다.

**이유:** 
- multiply 블렌드는 흰색(1)을 곱하면 원래 값이 유지되고, 검은 선(0)을 곱하면 0이 되어 선이 그대로 합성됩니다. 결과적으로 흰 배경은 투명하게 사라지고 각 공종의 선만 겹쳐 보이는 효과를 냅니다. screen이나 단순 opacity도 시도했지만 흰 배경이 불투명하게 남거나 선이 흐릿해지는 문제가 있었습니다.

## 5. 컴포넌트 설계

### 5-1. 탐색 로직 중앙화 (`useDrawingNavigation`)

도면/공종/리비전 선택 시 필요한 부수 효과(뷰 리셋, 최신 리비전 자동 선택 등)를 `useDrawingNavigation` 훅에 집중시켰습니다.

```
navigateToDrawing(id)
  → selectDrawing(id)       // drawingStore
  → resetView()             // viewStore

navigateToDiscipline(key)
  → selectDiscipline(key)   // drawingStore
  → selectRevision(latest)  // 최신 리비전 자동 선택
```

**이유:** 사이드바 건물 목록, 폴리곤 클릭 핸들러, 브레드크럼 등 여러 컴포넌트에서 동일한 탐색 동작이 필요합니다. 초기에는 각 컴포넌트에서 직접 selectDrawing() + resetView()를 함께 호출했는데, 뷰 리셋을 누락한 컴포넌트가 생기면서 "건물 전환 후 이전 줌이 그대로 남는" 버그가 있었습니다. 훅으로 올린 이후 누락 가능성이 없어졌습니다.

### 5-2. 폴리곤 레이어 계산 분리 (`usePolygonLayers`)

어떤 폴리곤을 그릴지 결정하는 로직을 `PolygonOverlay` 컴포넌트에서 분리하여 별도 훅으로 추출했습니다.

```
CASE 1: selectedDrawingId === '00'  → 건물 위치 폴리곤
CASE 2: !selectedDisciplineKey      → 공종별 영역 폴리곤
CASE 3: discipline.hasRegions       → 구역(A/B) 폴리곤
CASE 4: 기본                         → 리비전/공종 폴리곤
```

`useMemo`로 의존 상태가 변경될 때만 재계산하여 불필요한 SVG 재렌더링을 방지합니다.

**고민했던 대안:**
- 컴포넌트 내부 조건 분기: 처음에는 PolygonOverlay.tsx 내부에 조건 분기를 모두 작성했습니다. 렌더링 로직과 계산 로직이 한 파일에 있어 처음에는 편했지만, CASE가 4개로 늘어나자 컴포넌트가 100줄을 넘기면서 어떤 조건에서 어떤 폴리곤이 나오는지 파악하기 어려워졌습니다. 훅으로 분리하고 useMemo를 적용한 이후, 컴포넌트는 순수하게 렌더링만 담당하게 됐습니다.

## 6. UX 설계 결정

### 6-1. 공종별 고유 색상 시스템

공종마다 일관된 색상 팔레트를 정의하여, 사이드바 탭·폴리곤 오버레이·오버레이 패널 태그에 동일한 색상을 사용했습니다.

```typescript
const DISCIPLINE_COLORS = {
  건축: { dot: '#60a5fa', ... },   // 파란 계열
  구조: { dot: '#fb923c', ... },   // 주황 계열
  소방: { dot: '#f87171', ... },   // 빨간 계열
  ...
}
```

색상 일관성을 통해 사용자가 공종을 빠르게 식별할 수 있습니다.

### 6-2. 도면 전환 시 뷰 리셋

건물(도면)을 전환할 때 줌 배율과 패닝 오프셋을 초기값으로 리셋합니다. 이전 도면에서의 줌/패닝 상태가 다음 도면에 그대로 적용되면 사용자가 도면을 찾지 못하는 상황을 방지합니다.

### 6-3. 사이드바 내부 너비 고정

사이드바가 닫힐 때 내부 컨테이너 너비를 `264px`로 고정하여, 닫히는 애니메이션 중 텍스트가 줄바꿈되거나 레이아웃이 깨지는 현상을 방지합니다.

```tsx
// 외부: 너비 트랜지션으로 열고 닫기
<aside style={{ width: sidebarOpen ? '264px' : '0px', transition: '...' }}>
  {/* 내부: 항상 264px로 고정 → 찌그러짐 없음 */}
  <div style={{ width: '264px' }}>
    ...
  </div>
</aside>
```

## 7. 개선 방안 사항 

### 7-1. 리비전 비교 이미지 렌더링

**개선 사항:**
- 비교 모드에서 compareRevision의 이미지를 메인 도면 위에 반투명 오버레이로 표시해야 합니다. 상태 관리(drawingStore.compareRevision)와 UI 골격(노란 테두리 가이드라인)은 구현했지만, 실제 이미지를 렌더링하는 부분을 마무리하지 못했습니다.

**시도한 방법:**
- 비교 이미지를 다중 공종 겹쳐보기(overlayImages)와 동일한 방식으로 렌더링하려 했습니다. 그런데 두 기능의 목적이 다릅니다. 공종 겹쳐보기는 "같은 도면 위 다른 공종"을 보는 것이고, 리비전 비교는 "같은 공종의 다른 버전"을 비교하는 것입니다. 비교 이미지에는 별도의 투명도 조절 UI가 필요한지, 현재 도면을 반으로 나눠 보여주는 Split View 방식이 더 적합한지에 대한 UX 결정이 먼저 필요했고, 시간 내에 결론을 내지 못해 이미지 렌더링만 남겨두게 됐습니다.

**시도 가능한 대안:**
| 방식 | 장점 | 단점 |
|------|-----------|------------------------|
| 방식장점단점Opacity Overlay (현재 겹쳐보기와 동일) | 구현 단순, 전체 도면 동시 조회 | 선이 겹쳐 읽기 어려울 수 있음 |
| Split View (좌우 슬라이더) | 직관적 비교, 도면 선이 겹치지 않음 | 구현 복잡, 슬라이더 인터랙션 추가 필요 |
| 변경 영역만 diff 하이라이트 | 변경점을 즉시 파악 가능 | 이미지 픽셀 diff 연산 필요, 서버 처리 필요할 수 있음 |

### 7-2. imageTransform 미적용

**개선 사항:** 
- 메타데이터에 공종·리비전별 imageTransform(x, y, scale, rotation)이 정의되어 있는데, 현재 캔버스 렌더링에서는 이 값이 완전히 무시됩니다. 도면 이미지가 전체 배치도 상의 정확한 위치와 각도로 표시되어야 할 때 이 변환이 필요합니다.

**시도한 방법:**
<img> 태그에 직접 style={{ transform: '...' }}을 적용하는 방식을 시도했습니다. 그런데 현재 DrawingCanvas는 최상위 div에 CSS transform으로 줌/패닝을 구현하고 있어, 이미지 자체에 추가 transform을 적용하면 두 transform이 중첩(transform multiplication)되어 의도한 위치로 계산하기가 복잡해졌습니다. 변환 행렬을 직접 합성하는 방식도 고려했으나 시간이 부족해 일단 기본 렌더링을 우선했습니다.

**시도 가능한 대안:**
imageTransform을 CSS transform으로 직접 변환하되, 캔버스 transform과 별개로 이미지 컨테이너에만 적용하는 방법. 줌/패닝 transform과 이미지 위치 transform을 다른 DOM 레벨에서 분리하면 중첩 문제를 피할 수 있습니다. 다만 현재 아키텍처와 전환 비용이 크기 때문에 즉각 적용은 어려워보입니다.

### 7-3. 드래그 중 캔버스 밖으로 나갈 때 드래그 stuck

**개선 사항:** 
- 마우스를 빠르게 드래그하면 커서가 캔버스 영역 밖으로 나가고, 이때 onMouseLeave로 isDragging이 false로 리셋됩니다. 마우스 버튼을 여전히 누른 채로 다시 캔버스에 진입하면 드래그가 작동하지 않습니다.

**시도한 방법:**
onMouseLeave를 onMouseUp과 동일하게 처리하는 방식으로 구현했습니다. 단순하고 빠르게 처리할 수 있었지만 UX가 어색해졌습니다.

**시도 가능한 대안:**
```tsx
typescriptuseEffect(() => {
  const handleMouseUp = () => setIsDragging(false)
  window.addEventListener('mouseup', handleMouseUp)
  return () => window.removeEventListener('mouseup', handleMouseUp)
}, [])
```
- window 레벨에서 mouseup을 감지하면 캔버스 밖에서 버튼을 떼어도 드래그가 정상 종료됩니다. 구현 방법을 알고 있었지만 다른 기능 구현에 우선순위를 두면서 수정하지 못했습니다.
